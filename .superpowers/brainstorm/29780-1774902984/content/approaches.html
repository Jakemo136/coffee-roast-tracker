Implementation Approach

How should we structure the build? All three deliver the same features — the difference is sequencing and risk.

A

Vertical Slices (Recommended)

Build one complete page at a time — schema changes, GraphQL operations, and UI together. Start with Dashboard (the hub), then Roast Detail, Upload Modal, Bean Library, Compare, Settings.

Pros

  • Working app after each slice — demo-able throughout
  • Each page is independently testable
  • Schema changes land just-in-time (no unused DB columns sitting around)
  • Natural dependency order — Dashboard first means you see data flowing immediately

Cons

  • May need minor refactors as later slices reveal shared patterns
  • Design tokens / shared components might evolve as you go
B

Layer-by-Layer

All schema changes first (rating, flavors, off-flavors, bean metadata), then all GraphQL operations, then all UI components and pages.

Pros

  • Complete data model upfront — no migrations mid-build
  • GraphQL layer is fully defined before UI touches it

Cons

  • Nothing visible until the UI layer — long feedback loop
  • Harder to parallelize work
  • Schema decisions made without UI feedback tend to need revision
C

Foundation + Verticals

Small foundation phase first: update design tokens to Direction C palette, build shared components (StarRating, FlavorPill, Modal), do ALL schema migrations. Then vertical slices for each page.

Pros

  • Shared components are consistent from day one
  • Schema is settled before any UI work
  • Pages go faster once the foundation is in place

Cons

  • Foundation phase has no visible output — feels slow to start
  • Shared components designed in isolation may not fit perfectly

💡 My Recommendation: A (Vertical Slices)

The backend is already built and tested. We're primarily adding new schema fields + building the full frontend. Vertical slices let us see real progress after each page, and the schema changes are small enough (rating field, flavor tables, bean metadata columns) that they don't need a dedicated phase. Shared components like StarRating and FlavorPill will naturally emerge from the first pages that need them — Dashboard and Roast Detail — and be reusable by later pages.
