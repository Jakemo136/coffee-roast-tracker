1. Flavor Descriptor Library — Starting Set

Where should the default flavor descriptors come from?

A

SCA Flavor Wheel (curated subset)

~40-50 descriptors from the standard specialty coffee flavor wheel. Covers fruity, nutty, chocolate, floral, spice, etc. Familiar to anyone in specialty coffee.

B

Minimal starter set

~15-20 most common descriptors (chocolate, caramel, citrus, berry, nutty, floral, etc.). Less overwhelming, user builds up their own library over time.

C

Full flavor wheel (~100+)

Comprehensive set covering all SCA categories and sub-categories. More complete but potentially overwhelming in the UI.

2. Flavor / Off-Flavor Storage

How should flavors and off-flavors relate to roasts in the data model?

A

Tag-style (simple join)

A FlavorDescriptor table + a RoastFlavor join table. Each roast has many flavors, each flavor has many roasts. Clean and simple. No intensity or ordering.

B

Weighted tags (join with intensity)

Same as A, but the join table includes an intensity field (1-3 or 1-5). Lets you say "strongly chocolate, mildly citrus." More data, slightly more complex UI.

3. Rating — Where Can You Set It?

You mentioned rating lives on the roast detail view. Should it also be editable inline from the dashboard table?

A

Detail view only

Rating is displayed as read-only stars in the dashboard table. Click into a roast to set/change it. Keeps the table simple.

B

Inline + detail

Clickable star rating directly in the dashboard table row, plus on the detail view. Faster workflow — rate a roast without navigating away.

4. Roast Detail View — Layout Direction

The roast detail page needs to show: chart, metadata, tasting notes, flavors, off-flavors, rating, and profile actions. What layout feels right?

A

Single column, scrolling

Chart at top, then metadata cards, then flavor/notes sections stacked below. Simple, mobile-friendly, everything in reading order.

B

Two-column: chart left, metadata right

Chart takes ~60% width with metadata/phases sidebar. Flavors and notes below in full width. Makes good use of wide screens, collapses to single-column on mobile.

C

Tabbed sections

Chart always visible at top. Tabs below for: Overview | Tasting | Notes | Profile. Reduces scroll, but hides information behind clicks.

5. Upload Flow

Currently uploadRoastLog parses a .klog, creates the roast, and matches to a bean. What should the UX feel like?

A

Drag-and-drop → confirm

Drop zone on a dedicated upload page. After parsing, show a preview card with extracted metadata. User confirms bean match and adds notes/rating before saving.

B

Quick-upload modal from dashboard

Upload button on the dashboard opens a modal with drop zone. Faster for batch uploads — parse, confirm, save without leaving the table view.

C

Both (dedicated page + quick modal)

Dedicated page for first-time/detailed uploads. Quick modal for repeat users who just want to drop and go. More to build, but best UX.

6. Bean Library — How Much Detail?

Beans are currently a shared catalog with per-user notes. With URL scraping adding origin, process, elevation, and bag notes — how should the bean page work?

A

Simple list with expand

Table/card list of your beans. Click to expand inline with scraped details and linked roasts. No separate bean detail page.

B

Bean detail page

Full page per bean showing: scraped metadata, your notes, and a mini-dashboard of all roasts for that bean. Good for comparing how different roast profiles affect the same bean.

7. Comparison View Scope

The compare route exists but is a placeholder. For v1, what's the minimum viable comparison?

A

Overlaid roast curves only

Select 2-4 roasts, see their temperature curves overlaid on one chart with a legend. Core comparison tool for tuning profiles.

B

Curves + side-by-side metadata

Overlaid chart at top, plus a comparison table below showing key metrics (dev time, DTR%, temps, rating) side by side. More useful but more to build.
